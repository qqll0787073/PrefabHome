# PH-008D Signature Delivery Foundation

PH-008D adds an internal Signature Delivery Request foundation for `ready_to_send` Signature Packages. It prepares a request that can be queued internally for a future provider worker.

Queued means internal readiness only.

Queued does not mean sent, delivered, signed, completed, executed, or legally effective.

PH-008D does not implement real electronic signatures, signature images, PDF generation, DocuSign, Adobe Sign, signing links, email invitations, provider webhooks, sent/viewed/signed/declined/completed states, contract execution, legal effectiveness, payments, invoices, shipping, customs, notifications, milestones, or workflow automation.

## Migration

Migration: `supabase/migrations/0019_signature_delivery_foundation.sql`

PH-008D is additive over PH-008C. Migrations `0001` through `0018` must remain unchanged.

## Database Design

Tables:

- `public.signature_delivery_requests`
- `public.signature_delivery_recipients`
- `public.signature_delivery_events`

`signature_delivery_requests.delivery_number` is generated by the database as `SDL-YYYY-NNNNNN` from `public.signature_delivery_number_seq`.

Delivery statuses:

- `delivery_draft`
- `queued`
- `cancelled`

Provider key:

- `unconfigured`

No real provider identifiers, tokens, URLs, envelopes, webhook fields, signing links, or credentials are stored.

## Lifecycle

- Buyer creates one draft delivery request from their own `ready_to_send` Signature Package.
- Creation requires exactly two complete PH-008C signature participants.
- Recipient rows are derived from PH-008C participants.
- Buyer may queue a `delivery_draft` request internally.
- Queueing sets `queued_at` and creates a trusted event.
- Queueing performs no external calls and creates no sent state.
- Buyer may cancel a `delivery_draft` or `queued` request with a required reason.
- Cancellation sets `cancelled_at` and is terminal.

Lifecycle timestamp constraints:

- `delivery_draft`: `queued_at` null, `cancelled_at` null.
- `queued`: `queued_at` non-null, `cancelled_at` null.
- `cancelled`: `cancelled_at` non-null and cancellation reason required.

## Recipient Model

Recipients are immutable rows derived from Signature Package participants:

- Buyer signer
- Manufacturer signer

Signing order remains fixed:

- Buyer signer: order `1`
- Manufacturer signer: order `2`

Recipient `delivery_status` is `pending` only in PH-008D. There are no sent, viewed, signed, declined, or completed recipient states.

## Trusted RPCs

Public authenticated RPCs:

- `create_signature_delivery_request(package_uuid uuid)`
- `queue_signature_delivery_request(delivery_uuid uuid)`
- `cancel_signature_delivery_request(delivery_uuid uuid, reason_text text)`

Internal-only helpers and trigger functions explicitly revoke execution from `PUBLIC`, `anon`, and `authenticated`.

`can_access_signature_delivery_request(delivery_uuid uuid)` is a policy-only helper granted to `anon` and `authenticated` so PostgreSQL can evaluate recipient and event RLS policies. It does not grant table access by itself.

## Events

Trusted event values:

- `signature_delivery_created`
- `signature_delivery_queued`
- `signature_delivery_cancelled`

Events are database-generated, immutable, and undeletable. Impersonation and provider-secret metadata keys are stripped.

## RLS Summary

Anonymous users have no signature delivery access.

Authenticated read access:

- Buyer can read their own delivery requests, recipients, and events.
- Assigned Manufacturer can read assigned delivery requests, recipients, and events.
- Admin can read all delivery requests, recipients, events, and snapshots.

Writes:

- Direct table writes are revoked and blocked by triggers.
- Buyer lifecycle writes occur only through trusted RPCs.
- Manufacturer has read-only access.
- Admin has read-only access in PH-008D.

## Frontend

Buyer Portal:

- Shows Prepare Signature Delivery only for eligible `ready_to_send` packages.
- Shows Provider: Not configured.
- Supports Queue Internally from draft.
- Supports Cancel Request with a required reason.
- Queue confirmation states no provider is contacted, no email is sent, no signing link is created, and the Contract is not signed.
- Queued UI states the request is queued internally and not sent to a signature provider.
- Cancelled state is read-only.

Manufacturer Portal:

- Shows assigned delivery requests, recipients, and signing order.
- Read-only.
- Displays that no signature invitation has been sent.

Admin Portal:

- Shows all delivery requests, recipients, snapshots, and trusted events.
- Read-only.

## Provider Abstraction

`src/lib/signatureDelivery.ts` includes an interface for future provider adapters and an `unconfigured` no-op adapter only.

The PH-008D adapter does not:

- make network requests
- accept credentials
- generate signing URLs
- claim successful delivery

## Verification

Rollback-only SQL verification:

- `supabase/tests/signature_delivery_foundation_security.sql`

Expected behavior:

- Buyer can create, queue, and cancel only their own delivery requests.
- Manufacturer and Admin are read-only.
- Anonymous users are denied.
- Request number, snapshots, recipients, lifecycle timestamps, and events are database-derived.
- Direct table writes and event forgery are denied.
- Queued semantics remain internal-only.
