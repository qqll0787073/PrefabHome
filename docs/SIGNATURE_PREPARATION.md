# PH-008C Signature Preparation

PH-008C prepares accepted contracts for a future electronic-signature workflow. It does not implement electronic signatures, signature images, PDF generation, DocuSign, Adobe Sign, signing links, email invitations, provider webhooks, sent/viewed/signed/completed states, legal-effectiveness logic, payments, invoices, shipping, customs, notifications, production milestones, or workflow automation.

## Migration

Migration: `supabase/migrations/0018_signature_preparation.sql`

PH-008C is additive over PH-008B. Migrations `0001` through `0017` must remain unchanged.

## Database Design

Tables:

- `public.signature_packages`
- `public.signature_participants`
- `public.signature_package_events`

`signature_packages.package_number` is generated by the database as `SIG-YYYY-NNNNNN` from `public.signature_package_number_seq`. Application code never supplies signature package numbers.

Package statuses:

- `draft`
- `ready_to_send`

Participant roles:

- `buyer_signer`
- `manufacturer_signer`

Participant status is `pending` only in PH-008C. Later signature-delivery states are intentionally deferred.

## Lifecycle

- Buyer can create one draft signature package from their own accepted contract.
- Package creation requires an accepted contract review decision.
- Buyer can update only the Buyer signer participant while the package is draft.
- Assigned Manufacturer can update only the Manufacturer signer participant while the package is draft.
- Buyer can mark a draft package `ready_to_send` only after both participant records are complete.
- `ready_to_send` packages are immutable in PH-008C.
- Admin can read all signature packages, participants, events, and snapshots, but cannot mutate them.

`ready_to_send` means the package is prepared for a future signing-provider phase. It does not send the package, create a signing link, sign or execute the contract, or make the contract legally effective.

## Snapshot Model

Each signature package stores immutable snapshots captured at creation time:

- `contract_snapshot`
- `buyer_snapshot`
- `manufacturer_snapshot`
- `decision_snapshot`
- `signing_content_snapshot`

The signing content snapshot contains the contract title, governing law, contract terms, purchase order snapshot, quote snapshot, product snapshot, and line item snapshot from the accepted contract. Source contract or organization changes after package creation do not mutate the signature package snapshots.

## Trusted RPCs

Public authenticated RPCs:

- `create_signature_package(contract_uuid uuid)`
- `update_buyer_signature_participant(package_uuid uuid, full_name_text text, email_text text, title_text text)`
- `update_manufacturer_signature_participant(package_uuid uuid, full_name_text text, email_text text, title_text text)`
- `mark_signature_package_ready(package_uuid uuid)`

Internal-only helpers and trigger functions explicitly revoke execution from `PUBLIC`, `anon`, and `authenticated`.

`can_access_signature_package(package_uuid uuid)` is a policy-only helper granted to `anon` and `authenticated` so PostgreSQL can evaluate participant and event RLS policies. It does not grant table access by itself.

## RLS Summary

Anonymous users have no signature package access.

Authenticated read access:

- Buyer can read packages for their own contracts.
- Assigned Manufacturer can read packages assigned to their manufacturer profile.
- Admin can read all packages.

Writes:

- Direct table writes are revoked and blocked by triggers.
- Signature package writes occur only through the trusted RPCs.
- Signature package events are generated by trusted database functions.
- Admin cannot impersonate Buyer or Manufacturer participant updates.

## Frontend

Buyer Portal:

- Lists Buyer signature packages.
- Creates a package from eligible accepted contracts.
- Edits the Buyer signer participant while the package is draft.
- Marks the package ready after both signer records are complete.

Manufacturer Portal:

- Lists assigned signature packages.
- Edits the Manufacturer signer participant while the package is draft.
- Renders ready packages read-only.

Admin Portal:

- Lists all signature packages, participants, events, and snapshots.
- Remains read-only.

## Verification

Rollback-only SQL verification:

- `supabase/tests/signature_preparation_security.sql`

Expected behavior:

- Only the owning Buyer can create a package from an accepted contract.
- Duplicate packages are blocked.
- Package numbers are database-generated.
- Snapshots are immutable.
- Buyer and Manufacturer participant updates are scoped by role and ownership.
- Packages cannot be marked ready until both participants are complete.
- Ready packages are immutable.
- Anonymous users are denied.
- Admin read access works without mutation privileges.
- Direct event forgery and direct table writes are denied.
